## 二、文萃楼真实情况

### 2.1 地理位置
- **校区**: 北京理工大学良乡校区东校区（东北区域）
- **地址**: 北京市房山区良乡高教园区
- **性质**: 大型校园系统中的一个节点

### 2.2 楼宇结构（关键认知）
文萃楼不是"一栋楼"，而是：
> **多楼栋组合体（A/B/C/D/G/H/I/L等）**

不同楼栋承担不同功能：
- **教学功能**: 多媒体教室（双投影、阶梯教室等）、计算机考试机房（H栋）
- **科研与办公**: 设计认知与交互研究中心（C楼）、智算/信息中心（L栋）
- **活动与文化**: 展览/艺术活动空间

### 2.3 官方3D导航平台
- **平台地址**: https://wencuigis.info.bit.edu.cn/3dbuilding/#/building2
- **能力**: 提供1-10层完整平面图
- **本质**: 轻量级BIM + GIS融合的3D空间模型平台

## 三、导航难点的深度分析

### 3.1 四大核心难点

| 难点类型 | 具体表现 | 本质问题 |
|---------|---------|---------|
| **几何空间复杂** | 多楼栋拼接、非规则结构、内部连通关系复杂 | 三维空间 + 拓扑空间 |
| **语义空间复杂** | A栋≠C栋≠H栋，"机房""教务""报告厅"多种表达 | 坐标/名称/功能不一致 |
| **感知断裂** | 室外GPS可用，室内GPS失效 | Indoor Localization Gap |
| **认知vs地图** | 人走："进门→左转→上楼"，地图："坐标→坐标" | 认知空间 vs 几何空间不一致 |

### 3.2 关键认知突破
> 人类认知路径 ≠ 地图路径

这就是经典问题：**认知空间 vs 几何空间不一致**

## 四、系统架构设计

### 4.1 五层系统架构

```
感知层 (Perception)
  └─ GPS(室外) / IMU(运动) / Camera(视觉) / WiFi/蓝牙(辅助)
      ↓
信号层 (Signal)
  └─ 位置估计 / 滤波(Kalman/DSP) / 传感融合
      ↓
空间层 (Space Model)
  └─ 校园地图 / BIM(楼内结构) / 拓扑图(楼栋连接)
      ↓
智能层 (AI)
  └─ 路径规划 / 语义理解(A栋/机房) / 用户意图解析
      ↓
行为层 (Embodied Output)
  └─ 导航提示(转向/楼层) / AR指引(可扩展)
```

### 4.2 三层空间模型

```
物理空间 (Physical)  → 坐标 / 楼 / 路
语义空间 (Semantic)  → A栋 / 教务处 / 机房
行为空间 (Action)    → 左转 / 上楼 / 到达
          ↓
统一为：具身空间 (Embodied Space)
```

数学表达：
```
E = (Geometry, Topology, Semantics)
```

- **Geometry**: 房间 / 坐标
- **Topology**: 连接关系
- **Semantics**: A栋 / 教室 / 办公室

## 五、四次课教学设计

### 5.1 整体目标
> 基于文萃楼3D模型，构建一个"可运行的室内外一体化智能导航原型系统"。

### 5.2 学生最终成果
1. 一个**导航系统Demo**
2. 一份**系统设计报告**
3. 一次**现场演示（从A点导航到指定房间）**

### 5.3 四次课详细设计

#### 第1次课：问题建模（AI as Architect）
- **教学意图**: 将文萃楼从"一栋楼"转化为"一个CPS系统"
- **AI的作用**: 任务拆解、系统设计初稿
- **学生任务**: 用AI完成项目需求分析、系统模块划分、初步技术路线
- **产出**: 系统框架图（AI+人工）

#### 第2次课：空间建模 + 数据理解（AI as Modeler）
- **教学意图**: 理解"可计算空间"
- **AI的作用**: 自动解析楼层结构、生成拓扑图代码（Python/NetworkX）
- **学生任务**: 构建楼层拓扑图、房间连接关系
- **产出**: 可计算图结构（JSON格式）

#### 第3次课：路径规划 + AI决策（AI as Engine）
- **教学意图**: AI不是"画路线"，而是"做决策"
- **AI的作用**: 
  - 算法AI：A*/RL优化
  - LLM：用户指令理解、任务转化
- **学生任务**: 实现路径规划、语义解析（用LLM）
- **产出**: 可运行导航逻辑

#### 第4次课：系统集成 + 展示（AI as Full-stack）
- **教学意图**: 做一个"真的能用"的系统
- **AI的作用**: UI生成（Gradio/Web）、自动生成报告、自动生成演示脚本
- **产出**: 完整导航Demo演示

### 5.4 Demo预期效果
```
输入：我要去文萃楼D402
输出：
→ 进入东门
→ 左转
→ 上4层
→ 到达
```

## 六、核心技术方案

### 6.1 技术分层架构

#### 1. 空间层
解决"楼是什么样"的问题
- 1–10层平面图
- 房间、走廊、楼梯、电梯等元素
- 楼层间连接关系

#### 2. 语义层
解决"用户说的话对应哪里"的问题
- 房间号 → D402
- 功能名 → 教务办公室
- 区域名 → 延河文化艺术馆

#### 3. 路径层
解决"从起点到终点怎么走"的问题
- 起点节点 → 终点节点
- 路径规划
- 路径转成自然语言指令

#### 4. 交互层
解决"用户怎么用"的问题
- 输入框：起点、终点
- 输出：路径图 + 文本导航

### 6.2 数据结构建议（JSON格式）

```json
{
  "floor": 4,
  "nodes": [
    {"id": "D402", "type": "room", "x": 120, "y": 80, "name": "D402"},
    {"id": "corridor_1", "type": "corridor", "x": 100, "y": 80},
    {"id": "stairs_4A", "type": "stairs", "x": 60, "y": 30}
  ],
  "edges": [
    ["D402", "corridor_1"],
    ["corridor_1", "stairs_4A"]
  ]
}
```

### 6.3 语义映射方案

| 用户输入 | 系统目标 |
|---------|---------|
| D402 | D402 |
| 教务办公室 | D402 |
| A706 | A706 |
| 艺术馆 | C一层某区域 |

**实现方式**:
- 方案B1：规则匹配（稳定、简单、适合首版）
- 方案B2：LLM辅助语义解析（更智能、更符合AI教学主题）

**建议**: 首版采用"规则为主，LLM增强"的方案

### 6.4 路径规划方案
- **推荐算法**: Dijkstra / A*
- **理由**: 问题是楼宇拓扑图上的最短路径，不需要复杂强化学习

### 6.5 定位方案建议

> **关键决策**: 第一轮课堂项目，不做完整实时定位，只做"已知起点"的导航系统

**定位方式**:
- 方式A：手动选择（最稳定）
- 方式B：二维码扫码定位（技术难度低、现场演示强）
- 方式C：自动定位（后续再加）

## 八、技术底座方案

| 模块 | 推荐实现方式 | 理由 |
|-----|------------|------|
| **定位方案** | 二维码扫码 (QR Code) | 绕过复杂的室内高精度定位，直接获取"节点ID" |
| **空间数据** | JSON 拓扑图 | 易于AI解析，方便4次课内快速构建 |
| **算法语言** | Python | 与AI交互最友好，适合算法快速原型 |
| **前端** | H5 / Streamlit / Gradio | 开发成本低，课堂展示效果好 |
| **后端** | Python Flask / FastAPI | 返回路径JSON |
| **展示端** | 钉钉 H5 微应用 | 利用现有平台，降低开发门槛 |

## 十、实现原则（重要）

### 10.1 三条核心原则

1. **先做"能用"，再做"高级"**
   - 不要一开始追求：实时定位、多模态大模型、三维重建、端到端Agent
   - 先把"输入目标—输出路径"做扎实

2. **规则系统打底，AI负责增强**
   - 这样才稳定
   - 避免"AI看起来很聪明，但系统根本跑不起来"

3. **先把空间模型做清楚**
   - 项目成败的关键：文萃楼空间怎么被表示为机器可以计算的结构

### 10.2 技术路线建议

**第一步**: 选定一个最小区域，整理空间数据  
**第二步**: 建立房间/功能语义映射  
**第三步**: 实现图搜索路径规划  
**第四步**: 做一个可演示界面  
**第五步**: 再加入LLM增强解释能力

## 十一、教学创新亮点

### 11.1 三大创新点

**创新1: AI从"工具"变成"参与者"**
- 传统：学生写代码
- 现在：AI参与建模、设计、编码、报告

**创新2: 任务驱动（不是知识驱动）**
- 传统：讲DSP → 做作业
- 现在：解决导航问题 → 学DSP

**创新3: 具身化教学（核心优势）**
- 学生不是学算法
- 而是让系统"在楼里走"

### 11.2 课程总结表述
> 本课程基于真实空间场景，构建"AI参与的项目式教学模式"，通过引入大模型辅助任务建模、算法设计与系统实现，使学生在完成具身导航系统开发过程中，掌握从信号处理到智能决策的完整技术链条。
